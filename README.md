# LoopSession — P2P オンラインルーパー

サーバーレス P2P（WebRTC・手動シグナリング）でつながるオーディオルーパー。
縦長の DAW 風 UI。音の入り口は3つ:

1. **オーディオ I/F のライン入力**（ギター・マイク等）
2. **プラグイン音源 (WAM)** — アプリ内で Web Audio Modules のインストゥルメントを
   ホストし、MIDI キーボード / PC キーボードで演奏
3. **DAW のプラグイン音源** — DAW 出力をループバック/仮想デバイス経由で取り込み

公開版: https://kimama-2005.github.io/LoopSession/

モードは2つ（上部タブで切替）:

- **LOOPER** — 録音完了したループを転送・グリッド同期再生するメインモード
- **LIVE** — ルーパーを使わず、演奏をそのまま相手へ送り続けて
  1〜2小節遅れでグリッドに整列再生する遅延合奏（NINJAM方式・旧版の後継）

## 仕組み

- リアルタイム音声は送らない。**録音が完了したループ (Int16 PCM) を DataChannel で転送**し、
  受信側が共有クロックのテンポグリッドに位相を合わせて再生する。
  ネットワーク遅延は「ループが1周遅れて届く」ことにしかならず、演奏はズレない。
- 接続は**スター型**: 全員がホストと直結し、ゲスト間はホストが中継する。
  共有クロックはホストの `performance.now()` が基準。ゲスト側が
  ping/pong（NTP 方式・最小 RTT 採用）でオフセットを合わせる。
  LIVE 音声チャンクは自己記述ヘッダ（送信者ID・レート・遅延小節入り）なので
  中継・複数送信者・途中参加に耐える。
- 録音は次の小節頭から固定長（1/2/4/8小節）でクオンタイズ。
  オーバーダブはレイヤー追加方式（UNDO で1枚ずつ取り消し可能）。
- 入力は AEC/NS/AGC をすべて無効化して開く（ライン入力・音源入力を色付けしない）。
- 録音のレイテンシ補正: `baseLatency + outputLatency` を自動加算し、
  「設定 > 録音補正」で手動オフセットを足せる。

## 使い方

1. `python serve.py` → http://localhost:5173 （Chrome/Edge。`python -m http.server` は
   .js を text/plain で返すため不可）
2. START SESSION → 音の入り口を用意する
   - **ライン入力**: INPUT でオーディオ I/F のデバイスを選択
   - **プラグイン音源**: 「音源」で OBXD / TinySynth / Faust Flute（または任意の
     WAM2 プラグイン URL）を読み込む。MIDI キーボードは自動検出、
     KEYS を押すと PC キーボードでも演奏可（A W S E D... / Z X でオクターブ）。
     GUI ボタンでプラグインのパネルを開ける
   - **DAW の音源**: DAW の出力を I/F のループバックや VoiceMeeter 等の
     仮想デバイスへルーティングし、それを INPUT で選ぶ
3. トラックの ● を押すと次の小節頭から録音開始（停止中なら1小節カウントイン付きで自動スタート）。
   再生中に ● でオーバーダブ。
4. P2P 接続（スター型・2人以上対応）:
   - **かんたん接続（推奨）**: ホストが「ルームを作る」→ 表示された**6文字コード**を
     参加者に伝える → 参加者はコードを入れて「参加」。これだけで自動で繋がる
     （人数分OK・ルームは開きっぱなしで追加参加も同じコード）。
     SDP交換は公開MQTTブローカー経由で自動化し、内容はルームコード由来の
     AES-GCM鍵で暗号化（コードを知らない相手には読めない）。
     ブローカーはシグナリングのみで、音声・データは従来どおりP2P直結
   - **手動接続（フォールバック）**: ブローカーに繋がらない環境用に
     従来のコピペ方式も「手動接続」に残してある
   - 参加者同士のデータ（ループ・LIVE音声・操作）はホストが自動で中継する
   - 接続すると既存のループ・テンポ・走行状態・参加者名簿も自動で同期される

録音したループが遅れて聞こえる場合は「設定 > 録音補正」をプラス方向に調整。
モニターは基本 I/F 直（ダイレクトモニター）推奨。MON ボタンはブラウザ経由（遅延あり）。

### レベル調整・ミックス

- **INPUT の GAIN**: ライン入力のゲイン（0〜200%）。録音・LIVE送信・モニターすべてに
  効くので、環境ごとの音量差はまずここで揃える（メーターが目安）。端末ごとに保存される
- **マスター音量**（設定）も端末ごとに保存される
- **トラック名**: 自分のトラックは名前をクリックして変更できる（全員に同期）
- **設定 > ミックス「ホストに従う」**: オンだとホストの音量/ミュート操作が
  全員に反映される（合奏のバランスをホストが一括管理）。オフにすると
  自分の環境だけで自由に調整できる。端末ごとに保存

### LIVE モード（遅延合奏）

LIVE タブ → P2P 接続した状態で「LIVE送信」を ON にすると、ライン入力＋音源の音が
連続送信され、相手側では指定小節（1/2）遅れてテンポグリッドに整列して鳴る。
ルーパーを使わない掛け合いセッション用。ネットワーク遅延の条件は
「片道遅延 < 遅延小節の長さ」（例: 100BPM 4/4 の1小節 = 2.4秒）でかなり緩い。
専用 DataChannel を使うためループ転送と併用しても混線しない。

### バウンス（アプリ内 WAV 録音）

トランスポートの **WAV** ボタンで Master 出力の録音を開始し、もう一度押すと
16bit ステレオ WAV がダウンロードされる（ファイル名は日時、上限約10分）。
ルーティング不要でループ・LIVE・音源モニターを含む最終出力がそのまま録れる。
メトロノームも Master に乗るので、不要なら CLK をオフにしてから録る。

### バウンス（Master を WAV 保存）

トランスポートの **WAV** ボタンで Master 出力の録音を開始（ボタンが赤く点滅・経過表示）、
もう一度押すと 16bit ステレオ WAV がダウンロードされる。ルーティング不要で
ループ・LIVE受信・プラグイン音源・メトロノームなど「聞こえている音」がそのまま入る。
ライン入力の生音を含めたい場合は MON をオンにする（オフだと自分の生音は Master を
通らないため入らない）。上限10分で自動保存。

### Master 出力を DAW に戻す

「設定 > 出力先」で Master の出力デバイスを変更できる（Chrome/Edge 110+ の
`AudioContext.setSinkId`）。VoiceMeeter Input などの仮想デバイスや
オーディオ I/F の別出力を選び、DAW 側でそれを入力にすれば
セッション全体をそのまま録音・加工できる。

## コード構成

- `src/engine.js` — 共有クロック・AudioContext・キャプチャバス・トランスポート・メトロノーム
- `src/capture-worklet.js` — 入力を約1024フレーム毎にフレーム番号付きで送る AudioWorklet
- `src/track.js` — ループスロット（固定長レイヤー録音・グリッド位相整列再生）
- `src/p2p.js` — 手動シグナリング WebRTC・クロック同期・ループ転送プロトコル・PCM 変換
- `src/instrument.js` — WAM2 プラグインホスト・WebMIDI・PC キーボード演奏
- `src/live.js` — LIVE モード（連続チャンク送信・グリッド整列再生）
- `src/signal.js` — かんたん接続（公開MQTTブローカー経由の自動シグナリング・AES-GCM暗号化）
- `src/main.js` — UI とプロトコルの配線
- `legacy/` — 旧プロトタイプ（MIDI 合奏 + 音声ストリーミング β）

録音経路: ライン入力とプラグイン音源は `captureBus` に合流し、同じワークレットで
キャプチャされる（同時に鳴らして一緒に録ることも可能）。プラグインの読み込み先は
WAM 公式サンプルのホスティング（mainline.i3s.unice.fr）で、外部サーバーのため
落ちている場合は読み込みに失敗する（その場合もライン入力側は影響なし）。

## 既知の制限

- ピア間のオーディオクロックの個体差により、長時間（数十分）回し続けると
  数十 ms 単位のずれが蓄積しうる（トランスポート再スタートでリセット）。
- STUN のみ。対称 NAT 同士では繋がらない場合がある（必要になったら TURN を追加）。
- PCM 無圧縮転送（2小節ステレオ 48kHz ≒ 1MB 弱）。将来 Opus 化の余地あり。
- スター型のためホストの回線負荷が人数に比例して増える（LIVE 併用時は特に）。
  実用目安は 3〜4人程度。ホストが落ちるとセッション全体が切れる。
